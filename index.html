Read Calls Data

The starting point is searching for, main tasks that most frequently occur. These tasks then form the basis for conversation scripts to be designed either by modifying pre-existing versions or by creating for the first time. Each conversation script is classified into sections based on related tasks that exist for, and are stored in a wider repository for re-use.

In data analysis, although schema rectification is of utmost importance if need be however there is an alternative approach which is somewhat easier and that is the ability to read or search for data what ever location or state that may be found in, the condition being solely how is relevant to the wider purport of analysis.

The performance log is an essential part of this approach, the approach being, Read record from calls data, Transform into a conversation script, Classify the conversation script inside a repository, Log the conversation script as per use to track performance and then, Analyse to check how each conversation script is performing, and improving.

Ethical Guidelines On Prompting For Conversation Design:

  1. Paste Existing Script Into A Copilot Chat And Include A Note That Want To Discuss Variations Or Improvements

  2. Outline Criteria In A Copilot Chat And Request Varying Output

Prompt 1 Sample:

[Script]

Can I request you to help me by providing variations of the above script?
Can we discuss improvements?
Based on the above script, my requirements are, [requirements]

Prompt 2 Sample:

What follows is the criteria,

[criteria]

Can you provide varying scripts based on the above-listed criteria?

AI is excellent at generating text, and so long as prompts are detailed, they remain effective in providing conversation scripts.

Conversation Script Core Components:

After Copilot has provided with output, the structure can then be re-shaped as desired.

Each conversation script generally ought to consist of an:

  1. Opener - suited to requirement.
  2. Main flow - guided by data source{s}.
  3. Endpoint - that is both, friendly & professional.

Conversation scripts can be copied and modified to suit the requirements of the newly found tasks, if the contents are closely matching, this can be a speedier process than prompting Copilot each time although is recommended Copilot is prompted each time prior to creating/modifiying conversation script.

Conversation Scripts Version Control:

The performance log is where snapshots of outdated versions of conversation scripts can be stored and although it itself does not innately reveal a problem with a conversation script however if due to an interaction or experience a problem is found and a version is updated then the performance log can act as a storage tool for the outdated conversation. Why this helps is, if the updated version is not up to standards after it's deployment or is in production, then this outdated version is still available for use until the new version is prepared.

When a conversation script has been updated, snapshot the pre-existing version by attaching to the record existing in the performance log even though now an updated version exists. It's infact a huge misconception that because, now a new standardised edition exists, the old one is of no value because if in the unforeseen future the newer version lags in performance then a quick referral to what pre-exists can help a great deal.

DIA Exemption – Low Risk

All outputs within this ecosystem are classified under a DIA Exemption – Low Risk profile.

This means:

  • data used is non‑sensitive and appropriate for Copilot consumption
  • knowledge artefacts are approved, governed, and auditable
  • AI usage remains compliant with internal and regulatory controls

The low‑risk designation enables broader Copilot usage while maintaining trust and accountability.
